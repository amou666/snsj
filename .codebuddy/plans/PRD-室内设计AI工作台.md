## 产品概述

基于 Next.js 全栈 MVC 架构的室内设计 AI 工作台，覆盖从需求采集到效果图生成、从选材报价到施工落地的全链路设计流程，集成用户认证、积分充值、项目管理和后台管理系统。

## 核心功能

### 业务流程合并：5 大工作台 + 2 个功能层

原 6 大 AI 工具 + 12 项新功能按数据流合并，减少重复操作，分析结果可在步骤间复用省积分。

#### 合并后的业务入口

| 入口 | 包含的原功能 | 积分计算 | 核心交互 |
|---|---|---|---|
| **平面图工作台** | 户型图识别+自动建模、黑白转彩色、彩色转3D、空间布局规划 | 每步独立(分析1/生成1) | 链式管道：上一步输出自动做下一步输入 |
| **实景化工作台** | 毛坯转实拍、平面转实拍、3D转实拍、多方案对比 | 分析1+生成1=2/次，多方案每风格+1 | 自动识别输入类型，选择风格批量生成 |
| **方案助手** | 需求采集、布局规划、选材推荐、预算估算 | 每步独立(分析1/生成1) | 串联管道：需求→布局→选材→预算，可只走部分 |
| **深化工作台** | 施工图生成、全景图/VR预览 | 分析1+生成1 / 仅分析1 | 从已有效果图继续深化 |
| **空间拆解** | 保持独立 | 分析1积分 | 仅分析，上传即出结果 |
| **AI 对话助手** | 全局悬浮面板 | 复用底层积分逻辑 | 自然语言触发任何工作台 |
| **一键出图** | 原套餐功能 | 自动累加链路积分 | 快捷按钮，非独立页面 |

#### 1. 平面图工作台（4合1）

```
上传平面图
  │
  ├── 自动检测: 黑白 or 彩色 or 户型图?
  │
  ├── [黑白] → 上色(生成1积分) → 彩色平面图
  │                                    │
  │                    ┌───────────────┼───────────────┐
  │                    ▼               ▼               ▼
  │              转3D(生成1)     布局规划(分析1)    直接实景化
  │                    │               │               │
  │                    └───────────────┴───────┬───────┘
  │                                            ▼
  │                                    跳转「实景化工作台」
  │
  ├── [彩色] → 直接进入上述分叉
  │
  └── [户型图] → 识别+建模(分析1+生成1) → 可编辑结构图
                                                    │
                                              布局规划/转3D/实景化
```

**合并收益**：4个独立页面 → 1个工作台，图片只上传一次，分析结果在步骤间流转。

#### 2. 实景化工作台（3合1 + 多方案对比）

```
上传图片(毛坯/平面/3D) 或 从平面图工作台流入
  │
  ├── AI自动识别输入类型（毛坯房/平面图/3D模型）
  │
  ├── 选择风格（现代/北欧/日式/新中式...）
  │
  ├── ☑ 对比模式：同时生成2-3种风格对比
  │   └── 分析1次 + 生成N次 = 1+N积分
  │
  └── 生成结果可：
      ├── 收藏/下载
      ├── 跳转「深化工作台」→ 施工图/全景
      └── 保存到项目
```

**合并收益**：3个工具页 + 1个对比页 → 1个工作台，自动识别输入类型无需用户选。

#### 3. 方案助手（4合1）

```
方式A: 从项目需求采集进入
方式B: 独立使用

  ① 填需求问卷 → AI分析(1积分) → 需求报告
                                        │
  ② 基于需求生成布局方案 → 分析(1积分) → 布局图
                                        │
  ③ 基于布局选材推荐 → 分析+生成(2积分) → 选材清单+替换图
                                        │
  ④ 基于选材算预算 → 分析(1积分) → 预算方案

  每步可单独使用，也可一键全链路(一键出图)
```

**合并收益**：需求采集+布局规划+选材+预算 → 1个助手，4步串联，结果自动传递。

#### 4. 深化工作台（2合1）

```
从实景化工作台的结果进入，或从项目文件选择效果图
  │
  ├── 生成施工图(天花/灯具/插座) → 分析+生成=2积分
  ├── 生成全景图/VR → 分析1积分
  └── 两者都做：3积分
```

**合并收益**：施工图+全景 → 1个工作台，都是效果图的下游处理。

#### 5. 一键出图（宏操作，非独立页面）

```
触发方式：
  - 平面图工作台的「一键出图」按钮
  - 方案助手的「生成全套方案」按钮
  - 项目管理中「快速出方案」操作

本质 = 自动串联：
  平面图工作台(上色+3D) → 实景化(2种风格) → 选材+预算
  总积分 = 自动累加，用户确认后执行
```

#### 6. 协作与分享（内嵌功能，非独立页面）

- 效果图贴图钉评论 → 在实景化工作台结果页内嵌
- 方案分享链接 → 在收藏/结果页「分享」按钮
- 邀请协作成员 → 在项目详情页「团队」标签

#### 3. 阶段详情与产出物

**① 客户建档**

- 录入客户资料 → 创建项目
- 首次沟通笔记（需求初步了解）
- 设置项目截止日期和优先级

**② 需求采集**

- AI 设计问卷（客户自助填写或设计师代填）
- 上传原始户型图、现场照片
- AI 自动生成《设计需求报告》
- 生成客户偏好标签

**③ 方案设计** （核心产出阶段）

- 生成布局方案（平面图工作台 → 布局规划步骤）
- 生成多套效果图（实景化工作台 → 多风格对比）
- 选材清单（方案助手 → 选材步骤）
- 预算方案（方案助手 → 预算步骤）
- 版本管理：v1 → v2 → v3... 记录每次修改

**④ 确认签约**

- 方案对比展示（实景化工作台对比结果）
- 客户确认方案 → 锁定最终版本
- 生成合同/报价单（可导出PDF）
- 首付款记录

**⑤ 施工跟进**

- 施工图包（深化工作台 → 施工图生成）
- 材料下单清单
- 进度汇报（定期拍照上传）
- 中期款申请
- 设计变更记录

**⑥ 完工归档**

- 完工照片
- 结案报告
- 尾款结算
- 客户评价
- 项目数据导出为案例作品集

#### 8. 数据模型（新增表）

```sql
-- 项目主表
CREATE TABLE projects (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,              -- 设计师（项目负责人）
  name TEXT NOT NULL,                    -- 项目名称
  client_name TEXT,                      -- 客户姓名
  client_phone TEXT,                     -- 客户电话
  client_wechat TEXT,                    -- 客户微信
  address TEXT,                          -- 项目地址
  building_name TEXT,                    -- 楼盘名称
  area REAL,                             -- 面积 (㎡)
  house_type TEXT,                       -- 户型 (如 "3室2厅")
  budget_min REAL,                       -- 预算下限
  budget_max REAL,                       -- 预算上限
  stage TEXT DEFAULT 'client_filing',    -- 当前阶段
  priority TEXT DEFAULT 'normal',        -- 优先级: high/normal/low
  deadline DATE,                         -- 截止日期
  source TEXT,                           -- 来源渠道
  tags TEXT,                             -- 偏好标签 (JSON)
  status TEXT DEFAULT 'active',          -- active/archived/deleted
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 项目成员
CREATE TABLE project_members (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,              -- 团队成员
  role TEXT DEFAULT 'designer',          -- owner/designer/assistant/viewer
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 项目产出物（文件管理）
CREATE TABLE project_assets (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  folder_path TEXT NOT NULL,             -- 虚拟路径如 "02_设计方案/_v1/效果图"
  filename TEXT NOT NULL,
  file_type TEXT,                        -- image/pdf/doc/other
  file_data TEXT,                        -- base64 或文件路径
  file_size INTEGER,
  version INTEGER DEFAULT 1,
  description TEXT,
  created_by INTEGER NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES projects(id)
);

-- 设计需求
CREATE TABLE design_requirements (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL UNIQUE,
  rooms TEXT,                            -- 房间列表 (JSON)
  style_preferences TEXT,                -- 风格偏好 (JSON)
  functional_needs TEXT,                 -- 功能需求 (文本)
  special_requirements TEXT,             -- 特殊需求
  family_members TEXT,                   -- 家庭成员 (JSON)
  lifestyle_info TEXT,                   -- 生活方式描述
  reference_images TEXT,                 -- 参考图 (JSON base64数组)
  ai_report TEXT,                        -- AI 生成的需求报告
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES projects(id)
);

-- 任务表
CREATE TABLE project_tasks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  assignee_id INTEGER,                  -- 负责人
  task_type TEXT,                       -- design_review/client_comm/material/construction/payment
  due_date DATE,
  priority TEXT DEFAULT 'normal',
  status TEXT DEFAULT 'todo',           -- todo/in_progress/review/done
  completed_at DATETIME,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (assignee_id) REFERENCES users(id)
);

-- 沟通记录
CREATE TABLE project_communications (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  channel TEXT,                         -- phone/wechat/visit/email/other
  summary TEXT NOT NULL,                -- 沟通概要
  detail TEXT,                          -- 详细内容
  action_taken TEXT,                    -- 采取的操作 (如"更新偏好标签")
  created_by INTEGER NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (created_by) REFERENCES users(id)
);

-- 付款记录
CREATE TABLE project_payments (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  milestone TEXT NOT NULL,              -- 里程碑名称
  amount REAL NOT NULL,
  status TEXT DEFAULT 'pending',        -- pending/paid/overdue
  paid_at DATETIME,
  payment_method TEXT,
  receipt TEXT,                         -- 收据凭证 (base64)
  notes TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES projects(id)
);
```

### 基础设施补充项（开发首阶段落地）

#### 1. 全局搜索（CTRL+K 快速调出）

- 顶部导航栏常驻搜索框，快捷键 `Ctrl+K`（Mac: `Cmd+K`）聚焦搜索
- 搜索范围：项目名、客户姓名、客户电话、楼盘地址
- 搜索结果按类型分组：项目（置顶）、客户、文件
- 选中结果直接跳转到对应页面
- 后端 API：`GET /api/search?q=关键词`（跨表联合搜索，LIMIT 10）
- SWR 缓存搜索结果（2分钟内相同关键词不走网络）

#### 2. API 限流保护

- 使用 `@upstash/ratelimit` 或内置 LRU 缓存实现，不依赖外部 Redis
- 限流规则：

| 接口范围 | 限制 | 超限响应 |
| --- | --- | --- |
| `/api/auth/*` | 每 IP 每分钟 10 次 | 429 Too Many Requests |
| `/api/tools/generate` | 每用户每分钟 3 次 | 429 + 提示"操作太频繁" |
| `/api/chat/*` | 每用户每分钟 10 条消息 | 429 |
| 其他 `/api/*` | 每用户每分钟 60 次 | 429 |


- 超限响应体：`{ error: "rate_limit", message: "操作太频繁，请稍后再试", retryAfter: 60 }`
- 前端 SWR fetcher 全局拦截 429，显示 Toast 提示

#### 3. AI 请求超时处理（60 秒硬限制）

- 后端发起 AI 调用时设置 `AbortController` 超时为 60000ms
- 超时流程：

```
AI 调用开始 → 60s 倒计时
├── 60s 内返回成功 → 正常处理（扣减积分/返回结果）
└── 60s 未返回 → AbortController 中断请求
├── 回滚事务（不扣积分）
├── 记录 generation_record(status='failed', error='timeout')
└── 返回 { error: "timeout", message: "请求超时，请重试" }
```

- 前端显示：骨架屏/加载动画持续 60s → 超时后显示「请求超时，请稍后重试」错误状态 + 重试按钮
- 超时时间可在后台 `model_configs.timeout_ms` 中按模型独立配置（但默认 60000）

### 关键设计决策

1. **SWR 缓存持久化**：窗口关闭前将 Map 序列化到 localStorage，刷新后恢复，实现"有缓存直接展示、无缓存等骨架屏"
2. **AI 对话流式输出**：Next.js Route Handler 原生支持 ReadableStream，SSE 推送 AI 文本流
3. **全景图**：使用 @google/model-viewer Web Component 嵌入，Three.js 托管于 Next.js 静态资源
4. **协作评论**：贴图钉通过 CSS position:absolute + 百分比坐标实现，与图片缩放联动
5. **一键出图套餐**：作为工作台的宏操作按钮，自动编排多个 AI 调用（分析布局 → 出效果图 → 选材 → 算预算），用户确认总积分后一键执行，非独立页面
6. **积分扣减事务性（失败不扣，统一原子计价）**：

- **统一计价**：分析操作 = 1 积分，生成操作 = 1 积分，不再按工具独立定价
- **注册不送积分**：新用户需购买积分或订阅套餐后获得月含积分
- **扣减时机**：仅在 AI 模型**成功返回有效结果后**才扣减积分，API 调用失败/超时/返回错误一律不扣
- **原子操作**：better-sqlite3 事务包裹，积分扣减 + 日志插入在同一个事务中完成，事务失败则回滚
- **幂等保护**：每个请求生成唯一 `request_id`，后端记录已处理的 request_id，防止重复扣减
- **流程**：请求 → 计算积分(has_analysis+has_generation) → 记账(预扣) → 调用 AI → 成功则确认扣减/失败则回滚预扣
- **日志完整**：每次扣减记录 `user_id/amount/balance_after/tool_key/request_id/model_used/status(success/failed)`

7. **API Key 安全保障**：

- **零暴露原则**：API Key **仅存储在后端 SQLite 数据库**，任何 API 响应中**不得返回** API Key 字段
- **前端零感知**：设计师前端**完全不知道 API Key 的存在**，所有 AI 调用走 `/api/tools/generate` 后端代理
- **后端代理**：前端请求 `/api/tools/generate` → 后端从 `model_configs` 表读取 API Key → 后端发起 AI HTTP 请求 → 后端返回结果给前端
- **模型配置接口安全**：后台 `/api/admin/model-configs` 的 GET 列表接口中，`api_key` 字段**默认不返回**（或返回脱敏后的 `sk-****`），仅在编辑时允许填写
- **环境隔离**：production 环境可额外从环境变量 `AI_API_KEY` 读取，覆盖数据库中的 Key（便于运维轮换）

8. **SQLite 跨平台**：部署时在 Linux 服务器安装编译工具重新编译，不提交 node_modules

### 性能与缓存策略

- SWR dedupingInterval: 2000ms，避免重复请求
- AI 对话流式：首字 500ms 内返回，后续持续推送
- LazyImage：IntersectionObserver + 占位色 + priority 属性控制首屏
- 图片上传：服务端 base64 存储（生成记录表），可选云存储
- API 响应：列表接口支持分页/搜索/排序，避免全量传输
- 预算计算：前端预计算 + 后端验证，双重校验

### 积分扣减原子流程（统一原子计价）

**核心原则**：分析 = 1 积分，生成 = 1 积分，注册不送积分

```
请求到达 /api/tools/generate
    │
    ├── 1. 校验 Token → 获取 user_id
    ├── 2. 生成唯一 request_id (uuid)
    ├── 3. 检查 request_id 是否已存在 → 已存在则返回缓存结果（幂等）
    │
    ├── 4. 根据工具配置读取 has_analysis / has_generation
    │   └── 计算总积分: totalCost = (has_analysis ? 1 : 0) + (has_generation ? 1 : 0)
    │
    ├── 5. 检查积分余额是否 >= totalCost
    │   └── 不足 → 返回 402 Insufficient Credits
    │
    ├── 6. 开启 SQLite 事务
    │   ├── 插入 credits_log(amount=-totalCost, type='pending')
    │   └── 不更新用户余额（预记账，实际未扣）
    │
    ├── 7. [if has_analysis] 调用 AI 分析模型 (nano-banana-2)
    │   ├── 成功 → 继续
    │   └── 失败 → 回滚事务，标记 log.status='failed'，返回错误
    │
    ├── 8. [if has_generation] 调用 AI 生图模型 (gpt-image-2-all)
    │   ├── 成功 → 继续
    │   └── 失败 → 回滚事务，标记 log.status='failed'，返回错误
    │
    ├── 9. AI 返回成功 → 提交事务
    │   ├── 更新 log.status='success'
    │   ├── 更新 users.credits -= totalCost
    │   └── 插入 generation_record
    │
    └── 10. 返回 { success: true, outputImage, creditsDeducted: totalCost }
```

**积分服务统一接口**：
```typescript
// lib/services/credit-service.ts
function deductCredits(userId: number, opts: {
  hasAnalysis: boolean;    // 是否包含分析步骤
  hasGeneration: boolean;  // 是否包含生成步骤
  toolKey: string;         // 工具标识
  requestId: string;       // 幂等 ID
}): Promise<{ success: boolean; creditsDeducted: number }>
```

### 全局原则

1. **一个页面上不超过 4 种字号**（通常是 30/14/12 或 24/14/12）
2. **最小可用字号 11px**（仅用于 Badge 角标），正常最小 12px
3. **行高 = 字号 × 1.5**（正文）/ **1.2**（标题）
4. **所有字号使用 Tailwind 或 CSS Variables**，禁止硬编码 px
5. **文字颜色遵守色彩层级**：主要信息 `#1F2937` > 次要 `#4B5563` > 辅助 `#9CA3AF`
6. **主标题一律左对齐**，统计卡片数值居中，表格文字左对齐
7. **强调内容用字重区分**，不用下划线（链接除外）

## Agent Extensions 使用计划

### Skill

- **skill-creator**: 在构建 UI 组件库时，将 Skeleton/LazyImage/Button 等可复用组件封装为 Skill，供后续项目直接引用
- **playwright-cli**: 在最终打磨阶段，对 6 个 AI 工具页面、后台管理页面、新增功能页面进行全端截图验证 UI 一致性
- **find-skills**: 在实现全景图/VR 功能时，查找是否有适合的技术方案或可复用组件
- **多模态内容生成**: 在实现 AI 对话助手和智能选材推荐时，用于图文混合内容的生成和展示

### SubAgent

- **code-explorer**: 在移植现有 6 大 AI 工具页面时，批量读取 Vue 组件中的 prompt 拼接逻辑和样式细节作为参考；在实现新增功能时，探索现有组件库和数据结构以便复用模式
